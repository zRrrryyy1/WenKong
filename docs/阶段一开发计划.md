## 一、项目概述

### 1.1 项目背景

WenKong 温控系统基于 STM32F103C8 + DS18B20 + PID 算法实现对加热设备的精确温度控制。现有 Android App 可以通过蓝牙（HC-05 SPP）与下位机通信，实时显示温度、目标温度、PWM 输出等数据。但 App 存在以下不足：

- 只能连接**已配对**的蓝牙设备，首次使用 HC-05 时需要跳出 App 去系统设置里配对
- 未适配 Android 12+ 的**运行时权限**，在 Android 12+ 设备上可能崩溃或无法扫描
- **自整定**（ATUNE）完成后，App 只能在日志里看到结果，没有 APPLY / IGNORE 交互按钮
- 自整定过程中的**进度**仅以文本数字显示，不够直观

### 1.2 开发目标

本阶段聚焦 4 个核心功能，补齐 App 的基础体验短板：

| 编号 | 功能 | 优先级 | 工作量预估 |
|------|------|--------|-----------|
| F-01 | 蓝牙设备发现扫描 | P0 | 4h |
| F-02 | Android 12+ 运行时权限 | P0 | 2h |
| F-03 | 自整定 APPLY / IGNORE 按钮 | P1 | 1h |
| F-04 | 自整定进度可视化 | P1 | 3h |

### 1.3 技术栈

- **语言**：Java (兼容现有代码)
- **最低 SDK**：21 (Android 5.0)
- **目标 SDK**：32
- **蓝牙协议**：经典蓝牙 SPP (RFCOMM)
- **UI**：原生 Android View 体系（无第三方 UI 库依赖）

## 二、功能详述

### F-01：蓝牙设备发现扫描

#### 2.1.1 现状问题

当前 `showDeviceList()` 只读取 `BluetoothAdapter.getBondedDevices()`，即系统已配对设备列表。HC-05/HC-06 蓝牙模块通常需要先扫描发现，用户点击后才配对连接。

#### 2.1.2 功能要求

1. 提供两种设备发现模式：
   - **已配对列表**（当前方式，保留）
   - **扫描发现**（新增，默认方式）
2. 点击标题栏弹出设备选择对话框时，默认执行蓝牙扫描
3. 扫描结果列表实时更新，显示：
   - 设备名称
   - MAC 地址
   - 信号强度指示（RSSI 强弱示意）
4. 点击未配对设备 → 自动配对 + 连接
5. 点击已配对设备 → 直接连接
6. 扫描超时时间：12 秒
7. 提供"重新扫描"按钮

#### 2.1.3 交互流程

```
用户点击标题栏
    ↓
弹出设备选择对话框
    ├─ 显示"正在扫描..." + 进度指示器
    ├─ 实时更新发现的设备列表（去重）
    ├─ 每发现一个设备追加到列表
    ├─ 12秒后自动停止扫描
    └─ 用户点击设备 → 停止扫描 → 连接
```

#### 2.1.4 技术要点

- 使用 `BluetoothAdapter.startDiscovery()` 进行异步扫描
- 注册 `BroadcastReceiver` 监听 `ACTION_FOUND` 事件
- 防重复添加：以 MAC 地址为 key 去重
- 扫描期间保持 `BluetoothAdapter.cancelDiscovery()` 在连接前调用
- 兼容 API < 23 和 >= 23 的权限差异

#### 2.1.5 数据结构

```java
class DiscoveredDevice {
    String name;        // 设备名称
    String address;     // MAC 地址
    short rssi;         // 信号强度
    boolean isBonded;   // 是否已配对
}
```

#### 2.1.6 验收标准

- [x] 能扫描到处于可发现模式的 HC-05/HC-06
- [x] 列表实时更新，不重复添加
- [x] 点击设备能成功连接
- [x] 扫描超时后自动停止
- [x] 未授权时提示开启权限

### F-02：Android 12+ 运行时权限

#### 2.2.1 现状问题

`AndroidManifest.xml` 已声明以下权限：

```xml
<uses-permission android:name="android.permission.BLUETOOTH" />
<uses-permission android:name="android.permission.BLUETOOTH_ADMIN" />
<uses-permission android:name="android.permission.BLUETOOTH_CONNECT" />
<uses-permission android:name="android.permission.BLUETOOTH_SCAN" />
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
```

但在 Android 12+（API 31+）上，`BLUETOOTH_CONNECT` 和 `BLUETOOTH_SCAN` 属于 **运行时权限**，必须在代码中主动请求，否则：
- 扫描时返回空列表
- 连接可能静默失败
- 部分设备直接抛 SecurityException

#### 2.2.2 功能要求

1. App 启动时检查并请求所需权限
2. 权限请求顺序：
   - Android 12+：`BLUETOOTH_SCAN` + `BLUETOOTH_CONNECT` + `ACCESS_FINE_LOCATION`
   - Android 10-11：`ACCESS_FINE_LOCATION`
   - Android 9 以下：无需运行时权限
3. 权限被拒绝时给出合理说明，引导用户开启
4. 权限被永久拒绝（勾选"不再询问"）时，引导用户去系统设置中开启
5. 扫描按钮/标题栏在权限未授予时不可用（灰色），并提示用户

#### 2.2.3 交互流程

```
App 启动 / 用户点击连接
    ↓
检查权限
    ├─ 已授权 → 进入扫描/连接流程
    ├─ 未授权 → 弹出系统权限对话框
    │       ├─ 用户允许 → 继续
    │       ├─ 用户拒绝一次 → 下次再请求
    │       └─ 用户永久拒绝 → 弹窗引导去设置
    └─ 需要解释时 → 先弹 Rationale 说明
```

#### 2.2.4 权限请求实现

```java
private static final int REQUEST_BLUETOOTH_PERMISSIONS = 100;

// 需要请求的权限列表（按 API 级别）
private String[] getRequiredPermissions() {
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.S) {
        return new String[]{
            Manifest.permission.BLUETOOTH_SCAN,
            Manifest.permission.BLUETOOTH_CONNECT,
            Manifest.permission.ACCESS_FINE_LOCATION
        };
    } else if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.M) {
        return new String[]{
            Manifest.permission.ACCESS_FINE_LOCATION
        };
    }
    return new String[]{};
}
```

#### 2.2.5 验收标准

- [x] Android 12 上首次启动弹出权限请求
- [x] 拒绝一次后下次还能再次请求
- [x] 永久拒绝后引导至系统设置
- [x] 权限不足时扫描/连接入口不可用
- [x] Android 9 设备上不请求运行时权限

### F-03：自整定 APPLY / IGNORE 按钮

#### 2.3.1 现状问题

下位机自整定完成后，会发送如下数据：
```
Temp:27.00  Set:27.00  PWM:0  Mode:Result  Kp:6.0  Ki:0.6  Kd:0.7  Conf:5
```

此时需要用户发送 `APPLY` 或 `IGNORE` 命令来确认或拒绝参数。但当前 App 只有 **ATUNE** 和 **ABORT** 两个按钮，用户必须在日志区看到结果后，手动在目标温度输入框输入 `APPLY` 来发送命令，交互很不直观。

#### 2.3.2 功能要求

1. 当检测到 `Mode:Result` 时，界面自动弹出 **确认对话框**
2. 对话框显示：
   - `Kp = 6.0`
   - `Ki = 0.6`
   - `Kd = 0.7`
   - 置信度星级（★ 1-5）
3. 对话框提供两个按钮：
   - **「应用参数」** → 发送 `APPLY`
   - **「忽略」** → 发送 `IGNORE`
4. 若用户不操作，对话框持续显示直至用户做出选择
5. ADDITIONALLY：在按钮区增加两个小按钮「应用」「忽略」（常驻），用户也可直接点击

#### 2.3.3 交互流程

```
下位机发送 Mode:Result
    ↓
App 解析数据
    ↓
弹出确认对话框
    ├─ "自整定完成！推荐 PID 参数："
    ├─  Kp: 6.0    Ki: 0.6    Kd: 0.7
    ├─  置信度：★★★★☆
    ├─  [应用参数]  [忽略]
    ↓
用户点击 → 发送 APPLY / IGNORE
    ↓
对话框关闭，Toast 提示操作成功
```

#### 2.3.4 验收标准

- [x] `Mode:Result` 时自动弹出确认框
- [x] 点击"应用参数"发送 `APPLY\r\n`
- [x] 点击"忽略"发送 `IGNORE\r\n`
- [x] 数据显示正确（Kp/Ki/Kd/置信度）
- [x] 界面增加常驻 APPLY / IGNORE 小按钮

### F-04：自整定进度可视化

#### 2.4.1 现状问题

自整定过程中，下位机会发送包含进度的数据，例如：

```
Mode:Baseline  Prog:45%
Mode:Heat  Half:3/10  TrackingMax:30.5
Mode:Cool  Half:4/10  TrackingMin:24.8
```

当前 App 只在日志区显示这些文本，用户无法直观了解当前整定阶段和进度。

#### 2.4.2 功能要求

1. 在温度大字下方增加一个 **自整定进度条区域**
2. 区域包含：
   - **进度条**（水平 ProgressBar），显示整体进度 0~100%
   - **状态文本**，显示当前阶段（"基线采集" / "加热阶段" / "冷却阶段" / "计算中" / "完成"）
   - **阶段描述**，显示半周期计数和跟踪极值
3. 仅在自整定进行中时显示该区域，正常 PID 模式隐藏
4. 进度条颜色随阶段变化：
   - Baseline：灰色
   - Heat（加热）：橙色
   - Cool（冷却）：蓝色
   - Complete：绿色

#### 2.4.3 布局示意

```
┌──────────────────────────────────┐
│         25.3°C                    │  ← 温度大字（已有）
│         Mode:PID                  │  ← 模式文本（已有）
├──────────────────────────────────┤
│  ████████████░░░░░░  65%         │  ← 进度条（新增）
│  加热阶段 (4/10)                 │  ← 状态文本（新增）
│  最高跟踪: 30.5°C                │  ← 详情文本（新增）
├──────────────────────────────────┤
│  目标温度  误差  PWM  ...         │  ← 数据表（已有）
└──────────────────────────────────┘
```

#### 2.4.4 数据解析规则

| 下位机数据 | Mode 值 | 进度条 | 状态文本 |
|-----------|---------|--------|---------|
| `Mode:Baseline  Prog:45%` | Baseline | 45% | "基线采集 45%" |
| `Mode:Heat  Half:3/10  ...` | Heat | 动态计算 | "加热阶段 3/10" |
| `Mode:Cool  Half:4/10  ...` | Cool | 动态计算 | "冷却阶段 4/10" |
| `Mode:Result  Kp:...` | Result | 100% | "整定完成！" |
| `Mode:Failed` | Failed | — | "整定失败"（红色） |

#### 2.4.5 进度计算公式

```
Baseline 阶段: progress = (已采集次数 / 20) * 100%
Heat/Cool 阶段: progress = 10% + (已完成半周期数 / AT_MIN_CYCLES) * 80%
Result/Failed: progress = 100%
```

#### 2.4.6 验收标准

- [x] 自整定时显示进度条区域
- [x] 进度随 `Prog` 数据实时更新
- [x] 阶段文本正确显示当前状态
- [x] 正常 PID 模式时进度条区域隐藏
- [x] 颜色随阶段变化

## 五、开发计划

### 5.1 开发顺序

建议按依赖关系排序：

```
F-02 权限  →  F-01 扫描  →  F-04 进度  →  F-03 APPLY/IGNORE
  ↓              ↓
（前置条件）   （独立开发）   （独立开发）   （依赖 F-04 的数据解析）
```

**推荐开发顺序**：F-02 → F-01 → F-04 → F-03

### 5.2 各任务时间估算

| 任务 | 预估工时 | 说明 |
|------|---------|------|
| F-02 权限请求 | 2h | 逻辑简单但需在不同 API 上测试 |
| F-01 蓝牙扫描 | 4h | BroadcastReceiver + 去重 + UI 实时更新 |
| F-04 进度可视化 | 3h | 布局 + 数据解析 + 颜色控制 |
| F-03 APPLY/IGNORE | 1h | 弹窗 + 按钮状态 + 命令发送 |
| 集成测试 + Bug 修复 | 2h | 整体联调 |
| **合计** | **12h** | |

### 5.3 测试要点

1. **权限测试**：
   - Android 12（API 31）模拟器：确认弹出权限请求
   - Android 9（API 28）模拟器：确认不请求运行时权限
   - 权限拒绝/允许/永久拒绝三种场景

2. **蓝牙扫描测试**：
   - HC-05 上电可发现模式 → App 能扫描到
   - 扫描列表去重验证
   - 超时停止验证

3. **自整定测试**：
   - 发送模拟串口数据验证进度条更新
   - APPLY/IGNORE 按钮状态切换
   - 弹窗交互

## 六、风险与应对

| 风险 | 影响 | 概率 | 应对方案 |
|------|------|------|---------|
| Android 12 权限模型变化 | F-02 失效 | 低 | 按官方文档实现，已在目标 API 32 上验证 |
| 某些 HC-05 固件版本不支持自动配对 | F-01 用户体验差 | 中 | 保留"已配对设备"列表作为备选 |
| 扫描过程中连接操作可能中断扫描 | F-01 连接失败 | 中 | 连接前确保 `cancelDiscovery()` |
| 多个 `Mode:Result` 连续发送 | F-03 弹窗堆叠 | 低 | 加防止重复弹窗的标志位 |
